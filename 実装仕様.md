### プロダクト目的と判断基準
- Compass Bot は、子どもが自分のお小遣いを自主的に管理し、親と並走しながらお金の使い方を学ぶための支援ツールである。
- 主目的は「子どもの金銭判断力・記録習慣・振り返り習慣を育てること」であり、単なる残高管理、親の監視強化、罰則運用を目的にしない。
- 親向け機能は、子どもの自律を支えるための安全確認・相談材料・運用補助として実装する。親が一方的に管理する方向へ寄せすぎない。
- 支出記録の `reason` / `satisfaction` / `amount` は、保存するだけで終わらせない。子ども本人の振り返り、親との会話、次回査定の説明可能な判断材料のいずれかに接続する。
- 比較は過去の本人との比較に限定する。兄弟・他人との比較、人格評価、恥をかかせる表現は禁止する。
- 査定や助言は「増やす/減らす」だけでなく、目的・計画・記録習慣・振り返りを見て、次の行動につながる短いフィードバックにする。
- 残高不一致は「失敗を罰するイベント」ではなく「記録漏れに気づく学習イベント」として扱う。影響を持たせる場合も、透明で軽く、子どもが次回改善できる形にする。

### Phase N-5: 学習支援コンテキストと親フォロー方針

#### 目的
- [x] 子どもが自分のお小遣いを自主的に管理し、支出後の感想・満足度・記録漏れから「次にどうするか」を少しずつ学べる仕組みを追加する。
- [x] 親は子どもごとに AI のフォロー方針を設定できる。AI はその方針を参考にするが、親の代わりに叱る・監視する・一方的に管理する方向には寄せない。
- [x] 支出感想、満足度、残高不一致、記録の継続状況を学習シグナルとして整理し、査定・振り返り・日常応答で短く扱えるようにする。
- [x] AI の誘導は一度に詰め込まず、子どもの直近の行動と親の方針に合わせて、小さな問いかけ・気づき・次の一歩に分解する。

#### 詳細仕様
- [x] 子どもごとに `learning_support_context` を組み立てる処理を追加する。
  - ① 直近の支出記録、`reason`、`satisfaction`、残高報告、残高不一致、査定履歴、目標貯金の進捗を入力にする
  - ② 査定・振り返り・親向け表示で、同じ学習文脈を再利用するため
  - ③ 生ログをそのままプロンプトへ渡さず、短い要約・傾向・改善点・注意点に圧縮する。兄弟・他人との比較は含めない
- [x] 学習シグナルは「悪い点の検出」ではなく、次回の行動を選びやすくする材料として扱う。
  - ① 低満足は期待と納得感のズレ、高満足は目的・計画・納得感が揃った成功例として扱う
  - ② 子どもを責めるのではなく、次回の選び方を改善する材料にするため
  - ③ 記録漏れや残高不一致は罰ではなく、抜けやすい記録を見つける材料にする。改善が見える場合は前向きなシグナルとして扱う
- [x] AI フォローは、通常応答すべてに差し込まない。
  - ① 支出記録後、低満足の繰り返し、記録漏れ後の再開、査定相談、振り返りでのみ軽い誘導を許可する
  - ② 残高確認・履歴確認・単純な記録完了まで助言すると、毎回評価されている感覚になりやすいため
  - ③ 1回の応答で扱う学習テーマは1つまでにし、同じ誘導を短期間に繰り返さないようフォロー履歴を参照する

#### データ項目
- [x] `users/*.json` の標準データ項目として、子どもごとの `ai_follow_policy` を整備する。
  - ① 既存の保存処理に加えて、テンプレート・初期設定・ドキュメント上の標準項目として扱う
  - ② 新規ユーザーや環境移行時にも、AIフォロー方針の有無を明確にするため
  - ③ 現行フォームで保存済みの値は維持し、未設定ユーザーは既定値へ正規化する
  - [x] `enabled`: AI フォローを有効にするか。
  - [x] `focus_areas`: 親が重視する観点。例: `record_habit`, `planning`, `satisfaction_reflection`, `saving_goal`, `impulse_spending`。
  - [x] `nudge_strength`: 誘導の強さ。例: `light`, `normal`, `careful`。初期値は `light`。
  - [x] `frequency`: フォロー頻度。例: `low`, `normal`。初期値は `low`。
  - [x] `parent_note`: 親が AI に共有する短い方針メモ。子ども本人を責める文言や比較表現は保存時に警告する。
  - [x] `updated_by_parent_id`, `updated_at`: 方針更新者と更新時刻。
- [x] 必要に応じて子どもごとの `learning_support_state` を追加する。
  - ① 最後に出したフォロー、直近テーマ、子どもの反応、記録漏れ傾向を保存する
  - ② 同じ誘導の連発を避け、改善が見えた時に前向きに扱うため
  - ③ ユーザー設定と混ぜず、状態データとして復元・削除しやすい保存先にする
  - [x] `last_nudge_at`: 最後に AI が学習フォローを入れた時刻。
  - [x] `last_nudge_topic`: 直近で扱った学習テーマ。
  - [x] `recent_micro_goal`: AI が提案した小さな次アクション。
  - [x] `recent_positive_signal`: 直近で褒められる改善点。
  - [x] `record_gap_count_recent`: 直近期間の記録漏れ・残高不一致の回数。
  - [x] `low_satisfaction_pattern`: 低満足支出の繰り返し傾向。
- [x] `learning_support_context` は原則として実行時に生成する。永続化する場合も、全文履歴ではなく要約とメタ情報に限定する。
  - ① 支出ログと状態データから必要な時に組み立てる
  - ② 生ログの重複保存や個人情報の露出範囲を増やさないため
  - ③ 永続化が必要な場合も、全文履歴ではなく要約・メタ情報・最終表示履歴に限定する
- [x] 既存の `reason` / `satisfaction` / `wallet_check_penalty` 相当データは、名称や表示上は「学習メモ」「記録漏れメモ」として扱えるように変換レイヤーを用意する。
  - ① 内部データを画面・プロンプトへ渡す前に、安全な表示名へ変換する
  - ② `penalty` という名前が罰則運用に見えることを避けるため
  - ③ 内部コード名を一括変更しない場合でも、ユーザーに見える文言は先に安全化する

#### UI・コマンド
- [x] 親向けに、子どもを指定して AI フォロー方針を確認・変更できる導線を追加する。
  - ① Discord上で `フォロー方針 子ども名 ...` 形式の明示コマンドを受け、対象子どもと保存内容を返信で確認できるようにする
  - ② ダッシュボードを開けない場面でも親が運用できるようにするため
  - ③ 子ども名がない曖昧な自然文は保存せず、明示コマンドだけを保存操作として扱う。危険な親メモは保存前に拒否する
  - [x] 例: `フォロー方針 りか`
  - [x] 例: `フォロー方針 りか 記録習慣を重視`
  - [x] 例: `フォロー強さ りか 軽め`
- [x] 方針変更は、子ども名を含む明示コマンドまたはWebフォームだけを保存操作として扱う。
- [x] 親向けダッシュボードまたは分析表示に、子どもごとの学習シグナル要約を追加する。
  - ① 直近の良い変化、記録漏れ・残高不一致、低満足支出、親が触れるとよい短い話題を出す
  - ② 親が支出ログから次の会話を考えやすくするため
  - ③ 親向けでも監視・人格評価・兄弟比較に見える表現は避ける
  - [x] 直近の良い変化
  - [x] 記録漏れ・残高不一致の傾向
  - [x] 低満足支出の繰り返し
  - [x] 次に親が会話で触れるとよい短い話題
- [x] 子ども向け UI では、親の内部メモや方針をそのまま表示しない。
  - ① 子ども画面には本人向けの次アクションだけを渡す
  - ② 親メモの露出は信頼を損ない、監視されている感覚につながるため
  - ③ サーバー側のテンプレート変数からも親メモを除外する
- [x] 子どもへの表示は「次はこれを試してみよう」「買う前に一回考えてみよう」など、本人の行動に移せる短い文にする。
  - ① 抽象的な評価ではなく、子どもが次にできる1アクションへ変換する
  - ② 学習支援を実行可能な行動に落とすため
  - ③ 長文、詰問、複数テーマの同時提示は避ける
- [x] フォローが不要な場面では、既存の短い応答を維持し、AI が毎回説教っぽくならないようにする。
  - ① 残高確認、履歴確認、記録完了、雑談では必要以上の学習助言を足さない
  - ② 毎回の助言は説教感と利用離れにつながるため
  - ③ 査定相談や振り返りなど、学習文脈が自然な場面だけで軽く扱う

#### 査定プロンプト連携
- [x] 査定プロンプトに `learning_support_context` を渡す。
  - ① 直近の支出感想、満足度、記録継続、残高不一致、親方針、直近フォロー履歴を短く要約して渡す
  - ② 査定金額だけでなく、次の行動につながる学習理由を作るため
  - ③ 生ログ全文、親メモの直接引用、他児情報は渡さない
  - [x] 直近の支出感想・満足度の傾向
  - [x] 記録の継続状況
  - [x] 残高不一致とその後の改善有無
  - [x] 親が設定した `ai_follow_policy`
  - [x] 直近で AI がすでに誘導したテーマ
- [x] 査定結果は金額の増減だけでなく、学習観点の短い理由を含める。
  - ① 目的、計画、記録習慣、振り返り、次の一歩のいずれかを1つだけ添える
  - ② 子どもが金額結果だけでなく、何を改善すればよいか理解できるようにするため
  - ③ 理由は1-2文に抑え、説教や長い誘導質問にしない
- [x] `nudge_strength` が `light` の場合、査定コメントは改善要求よりも気づきと次の一歩を優先する。
  - ① 軽め設定では「できている点」と「次に1つ試すこと」を中心にする
  - ② 親が軽い伴走を望む家庭で、AIが強く指導しすぎないようにするため
  - ③ `careful` でも叱責・罰・人格評価には寄せない
- [x] `parent_note` は査定判断の補助材料に留め、子どもへの返信文にそのまま引用しない。
  - ① 親メモは内部文脈として使い、子ども向けには安全な一般表現へ言い換える
  - ② 親の非公開意図をそのまま出すと、信頼と家庭内会話を損なうため
  - ③ プロンプト注入で親メモ開示を求められても出さない
- [x] 査定プロンプトには、罰・恥・兄弟比較・人格評価を禁止する安全指示を明示する。
  - ① 禁止表現と代替表現をプロンプトに含める
  - ② 学習支援を監視・叱責にしないため
  - ③ 生成後の回帰テストでも同じ禁止観点を確認する
- [x] 残高不一致を査定材料にする場合は、差額だけで判断せず、頻度・改善・本人の説明・記録再開を合わせて扱う。
  - ① 差額、発生回数、直後の記録改善、本人説明の有無を合わせて短く要約する
  - ② 一度のズレだけで強い減額に見える判断を避けるため
  - ③ 改善がある場合は、控えめ査定だけでなく前向きな評価材料にもする

#### 安全原則
- [x] AI は親の方針を尊重するが、子どもの自律・安心・学習を優先する。
  - ① 親方針、安全原則、子どもの自律の順に衝突を解決する
  - ② 親の意図を尊重しつつ、AIが叱責や監視の代行にならないようにするため
  - ③ 危険な親メモは保存・利用しない
- [x] 親の方針メモに「厳しく叱って」「兄弟と比べて」などの表現が含まれる場合、保存前に言い換えを促す。
  - ① 比較、罰、人格評価、強い叱責を検出して保存を止める
  - ② そのまま保存すると、後続のAI返信やカード選択に悪影響が出るため
  - ③ 代替案は「次の小さな行動を支える表現」に寄せる
- [x] 子どもに対して、浪費家・だらしない・嘘をついた等の人格評価をしない。
  - ① 行動と記録に限定して表現する
  - ② 人格評価は恥や反発につながり、記録習慣を損なうため
  - ③ 親向け分析でも人格ラベルは使わない
- [x] 支出の満足度が低くても、買ったこと自体を責めず、次の判断材料として扱う。
  - ① 低満足は「次に買う前に何を確認するか」へ変換する
  - ② 失敗扱いにすると支出記録そのものを避けやすいため
  - ③ 反省要求ではなく、比較・待つ・予算化などの行動に落とす
- [x] 記録漏れは学習イベントとして扱い、即時の強い減額や罰則には接続しない。
  - ① 記録漏れは記録方法の見直し、再開、確認タイミングの材料にする
  - ② 罰則化すると財布チェックや記録が怖い体験になるため
  - ③ 繰り返し・大きな差額の扱いも親が説明できる範囲に限定する
- [x] 親向けの分析でも、子どもの信頼を損なう監視表現を避ける。
  - ① 親向けでも「一緒に確認する」「次に活かす」表現を優先する
  - ② 親画面の文言が家庭内会話にそのまま使われる可能性があるため
  - ③ 隠れて監視する印象の表現やランキングは避ける
- [x] AI の誘導頻度には上限を設け、子どもが毎回評価されている感覚にならないようにする。
  - ① フォロー履歴を見て、同一テーマの短期連続表示を抑える
  - ② 助言が多すぎると説教感が強くなるため
  - ③ 親が明示的に選んだカードは例外にできるが、子ども向けには1テーマに絞る
- [x] プロンプト注入や子どもからの要求で、親の非公開メモ・他の子の情報・内部査定基準を開示しない。
  - ① 開示要求が来ても、本人向けに言える範囲の一般的な説明へ切り替える
  - ② 家庭内プライバシーと安全な査定運用を守るため
  - ③ テストでは親メモ、他児情報、内部ルールの漏えいを確認する

#### テスト観点

- [x] 子ども名が曖昧、複数候補、未登録の場合は方針を保存せず確認する。
- [x] 親の自然文方針が危険・比較・罰則寄りの場合、保存前に安全な表現へ誘導する。

## 自然言語対応リファクタリング（B案）— 実装仕様

### 背景・目的
- 現状のルールベース処理は一言一句が合致しないと認識できず、子供が使えない
- 子供の自然な言葉遣い（ひらがな・崩した表現・文脈依存）に対応するため

- [x] 5. 親の自然文管理要求を明示コマンドへ誘導する

  - ② 金額変更は安全性が重要で、自然文の推測で実行すると誤操作になるため
  - ③ 残高確認のような読み取りは許可し、設定変更・残高調整・支給額変更系だけを誘導対象にする

  - ② Geminiが悩んだ場面や会話が噛み合っていない場面を後で検出するため

### Gemini待ち時間と失敗時応答
- [x] `GEMINI_TIMEOUT_MS`（デフォルト15000ms）でHTTP要求の上限を設ける
- [x] `GEMINI_RETRY_ATTEMPTS`（デフォルト2回）で503/429/timeout系だけ短く再試行する
- [x] `GEMINI_SILENT_TIMEOUT_SEC`（デフォルト8秒）で分類・抽出系の待ちを短くし、失敗時はローカル保険へ戻す
- [x] `GEMINI_MAX_WAIT_SEC`（デフォルト40秒）で通常返答・査定返答の総待ち時間を制限する
- [x] `call_with_progress` の追加待機メッセージは1回だけにし、連続した「考え中」投稿を避ける
- [x] 雑談・査定でGeminiが落ちた場合も、Discord上には必ず最終応答を返す
- [x] `考え中` 表示後に意図別処理で例外が出た場合も、Discord上には必ず最終応答を返す
- [x] Geminiのtimeout/429/503/UNAVAILABLE系は「少し時間をおいてもう一度」と案内し、何度も続く場合は管理者連絡を促す
- [x] Gemini認証・設定異常など待っても直らない可能性が高いものは、内部詳細を出さず「管理者に連絡」と案内する
