### プロダクト目的と判断基準
- Compass Bot は、子どもが自分のお小遣いを自主的に管理し、親と並走しながらお金の使い方を学ぶための支援ツールである。
- 主目的は「子どもの金銭判断力・記録習慣・振り返り習慣を育てること」であり、単なる残高管理、親の監視強化、罰則運用を目的にしない。
- 親向け機能は、子どもの自律を支えるための安全確認・相談材料・運用補助として実装する。親が一方的に管理する方向へ寄せすぎない。
- 支出記録の `reason` / `satisfaction` / `amount` は、保存するだけで終わらせない。子ども本人の振り返り、親との会話、次回査定の説明可能な判断材料のいずれかに接続する。
- 比較は過去の本人との比較に限定する。兄弟・他人との比較、人格評価、恥をかかせる表現は禁止する。
- 査定や助言は「増やす/減らす」だけでなく、目的・計画・記録習慣・振り返りを見て、次の行動につながる短いフィードバックにする。
- 残高不一致は「失敗を罰するイベント」ではなく「記録漏れに気づく学習イベント」として扱う。影響を持たせる場合も、透明で軽く、子どもが次回改善できる形にする。

### Phase N-5: 学習支援コンテキストと親フォロー方針

#### 目的
- [ ] 子どもが自分のお小遣いを自主的に管理し、支出後の感想・満足度・記録漏れから「次にどうするか」を少しずつ学べる仕組みを追加する。
- [ ] 親は子どもごとに AI のフォロー方針を設定できる。AI はその方針を参考にするが、親の代わりに叱る・監視する・一方的に管理する方向には寄せない。
- [ ] 支出感想、満足度、残高不一致、記録の継続状況を学習シグナルとして整理し、査定・振り返り・日常応答で短く扱えるようにする。
- [ ] AI の誘導は一度に詰め込まず、子どもの直近の行動と親の方針に合わせて、小さな問いかけ・気づき・次の一歩に分解する。

#### 詳細仕様
- [ ] 子どもごとに `learning_support_context` を組み立てる処理を追加する。
  - [ ] 直近の支出記録、`reason`、`satisfaction`、残高報告、残高不一致、査定履歴、目標貯金の進捗を入力にする。
  - [ ] 生ログをそのままプロンプトへ渡さず、短い要約・傾向・改善点・注意点に圧縮する。
  - [ ] 兄弟・他人との比較は含めず、過去の本人との比較に限定する。
- [ ] 学習シグナルは「悪い点の検出」ではなく、次回の行動を選びやすくする材料として扱う。
  - [ ] 低満足度の支出が続く場合は、買う前の期待と買った後の納得感のズレとして扱う。
  - [ ] 高満足度の支出は、目的・計画・納得感が揃った成功例として扱う。
  - [ ] 記録漏れや残高不一致は、罰ではなく「どの記録が抜けやすいか」を見つける材料として扱う。
  - [ ] 記録改善が見える場合は、査定や振り返りで前向きな学習シグナルとして扱う。
- [ ] AI フォローは、通常応答すべてに差し込まない。
  - [ ] 残高確認・履歴確認・単純な記録完了メッセージでは原則短く返す。
  - [ ] 支出記録後、低満足の繰り返し、記録漏れ後の再開、査定相談、振り返りでのみ軽い誘導を許可する。
  - [ ] 1回の応答で扱う学習テーマは1つまでにする。
  - [ ] 同じ誘導を短期間に繰り返さないよう、直近の AI フォロー履歴を参照する。

#### データ項目
- [ ] `users/*.json` に子どもごとの `ai_follow_policy` を追加する。
  - [ ] `enabled`: AI フォローを有効にするか。
  - [ ] `focus_areas`: 親が重視する観点。例: `record_habit`, `planning`, `satisfaction_reflection`, `saving_goal`, `impulse_spending`。
  - [ ] `nudge_strength`: 誘導の強さ。例: `light`, `normal`, `careful`。初期値は `light`。
  - [ ] `frequency`: フォロー頻度。例: `low`, `normal`。初期値は `low`。
  - [ ] `parent_note`: 親が AI に共有する短い方針メモ。子ども本人を責める文言や比較表現は保存時に警告する。
  - [ ] `updated_by_parent_id`, `updated_at`: 方針更新者と更新時刻。
- [ ] 必要に応じて子どもごとの `learning_support_state` を追加する。
  - [ ] `last_nudge_at`: 最後に AI が学習フォローを入れた時刻。
  - [ ] `last_nudge_topic`: 直近で扱った学習テーマ。
  - [ ] `recent_micro_goal`: AI が提案した小さな次アクション。
  - [ ] `recent_positive_signal`: 直近で褒められる改善点。
  - [ ] `record_gap_count_recent`: 直近期間の記録漏れ・残高不一致の回数。
  - [ ] `low_satisfaction_pattern`: 低満足支出の繰り返し傾向。
- [ ] `learning_support_context` は原則として実行時に生成する。永続化する場合も、全文履歴ではなく要約とメタ情報に限定する。
- [ ] 既存の `reason` / `satisfaction` / `wallet_check_penalty` 相当データは、名称や表示上は「学習メモ」「記録漏れメモ」として扱えるように変換レイヤーを用意する。

#### UI・コマンド
- [ ] 親向けに、子どもを指定して AI フォロー方針を確認・変更できる導線を追加する。
  - [ ] 例: `フォロー方針 りか`
  - [ ] 例: `フォロー方針 りか 記録習慣を重視`
  - [ ] 例: `フォロー強さ りか 軽め`
- [ ] 方針変更は自然文だけで即確定しない。子ども名、重視観点、強さ、頻度を確認し、親の明示確認後に保存する。
- [ ] 親向けダッシュボードまたは分析表示に、子どもごとの学習シグナル要約を追加する。
  - [ ] 直近の良い変化
  - [ ] 記録漏れ・残高不一致の傾向
  - [ ] 低満足支出の繰り返し
  - [ ] 次に親が会話で触れるとよい短い話題
- [ ] 子ども向け UI では、親の内部メモや方針をそのまま表示しない。
- [ ] 子どもへの表示は「次はこれを試してみよう」「買う前に一回考えてみよう」など、本人の行動に移せる短い文にする。
- [ ] フォローが不要な場面では、既存の短い応答を維持し、AI が毎回説教っぽくならないようにする。

#### 安全原則
- [ ] AI は親の方針を尊重するが、子どもの自律・安心・学習を優先する。
- [ ] 親の方針メモに「厳しく叱って」「兄弟と比べて」などの表現が含まれる場合、保存前に言い換えを促す。
- [ ] 子どもに対して、浪費家・だらしない・嘘をついた等の人格評価をしない。
- [ ] 支出の満足度が低くても、買ったこと自体を責めず、次の判断材料として扱う。
- [ ] 記録漏れは学習イベントとして扱い、即時の強い減額や罰則には接続しない。
- [ ] 親向けの分析でも、子どもの信頼を損なう監視表現を避ける。
- [ ] AI の誘導頻度には上限を設け、子どもが毎回評価されている感覚にならないようにする。
- [ ] プロンプト注入や子どもからの要求で、親の非公開メモ・他の子の情報・内部査定基準を開示しない。

#### テスト観点

- [ ] 子ども名が曖昧、複数候補、未登録の場合は方針を保存せず確認する。
- [ ] 親の自然文方針が危険・比較・罰則寄りの場合、保存前に安全な表現へ誘導する。
- [ ] 支出感想・満足度・記録漏れ・残高不一致から `learning_support_context` が生成される。
- [ ] 低満足支出が続いた場合、AI は責めずに次回の買う前チェックへ誘導する。
- [ ] 高満足支出や記録改善がある場合、査定コメントで前向きな学習シグナルとして扱われる。
- [ ] フォロー頻度上限により、同じ子どもへ短期間に同じ誘導が連続しない。
- [ ] 残高確認・履歴確認・記録完了などの情報取得/記録操作では、不要なコーチングが混入しない。
- [ ] 査定プロンプトに `learning_support_context` と `ai_follow_policy` が渡り、返信文に親の非公開メモが直接出ない。
- [ ] 子どもからのプロンプト注入で、親メモ・他児情報・内部ルールが漏れない。
- [ ] 兄弟比較、人格評価、罰表現を含む返信が生成されないことを自動テストとプロンプト回帰テストで確認する。

### Phase N-6: 学習支援ダッシュボード再設計（会話カード化）

#### 位置づけ
- [ ] Phase N-5 の学習支援サマリーを暫定実装として扱い、Phase N-6 で会話カード方式へ置き換える。
  - ① 現行の `学習シグナル` / `フォローのヒント` は残しつつ、新しい `今週の会話カード` を主表示にする
  - ② 現行文言は似やすく、買った物・金額・理由・満足度を親の具体行動に変換しきれていないため
  - ③ 置き換え時は既存の親/子ダッシュボード表示を壊さず、旧サマリーは折りたたみの補助情報として残す
- [ ] 親向けダッシュボードを「監視・評価画面」ではなく「親子会話を始める画面」として再設計する。
  - ① 子どもごとに、根拠、伸ばしたい力、親の1問、今日の伴走をまとめたカードを表示する
  - ② 親が次に何を聞くか分からない状態を減らし、責めない会話へ接続するため
  - ③ 点数化、ランキング、兄弟比較、罰を連想させる文言は表示しない
- [ ] 子ども向け表示を `評価結果` ではなく `次の小さなチャレンジ` に変更する。
  - ① 子どもには1回1つ、5分以内にできる行動だけを提示する
  - ② 分析結果をそのまま見せるより、実行できる小さな行動にした方が自律につながるため
  - ③ 親メモ、親の心配、他の子との比較、内部判定理由は子ども画面に出さない
- [ ] AI は生ログから自由作文せず、ルールベースで選んだカードの言い換えだけに使う。
  - ① `learning_insights` がカード候補と根拠を決め、AI は年齢・口調・親方針に合わせて短く整える
  - ② 生ログ丸投げは、根拠の揺れ、個人情報露出、説教調の生成につながりやすいため
  - ③ AI出力は安全原則、親メモ非公開、ギャンブル・借金・高リスク投機禁止のルールを必ず通す

  - ② 単なる交渉ではなく、家庭で合意できる行動と学習機会へ接続するため
  - ③ 親が明示操作した場合だけ、臨時収入・役割報酬・増額条件を保存する。AI は金額を勝手に確定しない
  - [ ] 例: 「毎週の記録を続ける」「買う前に比較する」「目標貯金へ毎月いくら残す」「家庭内で決めた役割に取り組む」。

  - [ ] 少額低満足反復: 小さな衝動支出の積み上がりに気づくカードを優先する。
  - [ ] 高額高満足: よい判断の理由を言語化するカードを優先する。
  - [ ] 少額高満足: 少ない金額で満足できた成功例として扱う。

  - ② 親が重視する支援テーマと、画面に出るカードが一致していないと運用しづらいため
  - ③ 安全原則に反する親メモは重みに使わず、カード根拠が弱いテーマを無理に出さない
  - [ ] `focus_area=record_habit`: 記録再開、理由・満足度補足、記録タイミングを優先する。
  - [ ] `focus_area=planning`: 欲しいものリスト、買う前待機、月予算を優先する。
  - [ ] `focus_area=satisfaction_reflection`: 高額低満足、低満足反復、成功例の言語化を優先する。
  - [ ] `focus_area=saving_goal`: 目標貯金への影響、残り期間、今月残す額を優先する。
  - [ ] `focus_area=impulse_spending`: 少額反復、オンライン課金、買う前チェックを優先する。
  - [ ] `focus_area=income_balance`: 固定お小遣い、入金、支出比率、残す金額、役割と収入の関係を優先する。
- [ ] 親が `この子にはこうフォローしてほしい` をAIへ渡せる導線を整理する。

### 支出感想・残高不一致のあるべき扱い
- 支出感想・満足度は、低満足な支出を責めるためではなく、「買う前の期待」と「使った後の納得感」を比べる材料にする。
- 次回査定では、直近の支出記録から次のような学習シグナルを要約して渡す。
  - 低満足度の支出が同じ品目・同じ理由で続いているか
  - 高満足度の支出に目的・計画・学習への接続があったか
  - 記録が継続しているか、金額・理由・満足度が揃っているか
  - 残高不一致の原因が支出漏れか収入漏れか、直後に記録改善があったか
- 「ペナルティ」は強い罰として扱わない。実装上は `wallet_check_penalty` という名前が残っていても、プロダクト上は「記録漏れメモ」「監査メモ」に近い概念として扱う。
- 残高不一致の影響は段階的にする。
  - 初回・軽微: 帳簿修正と気づきのメッセージのみ
  - 繰り返し・大きな差額: 次回査定で臨時分を控えめにする材料として明示
  - 改善が見える場合: 次回査定で「記録改善」として前向きに扱う
- 金額を機械的に減額する即時ペナルティは原則採用しない。必要な場合でも親が明示的に確認できる形にし、子どもに理由を説明できることを条件にする。
- 画面・メッセージ上の文言も「罰」より「記録漏れ」「次に活かす」「一緒に確認する」を優先する。

## 自然言語対応リファクタリング（B案）— 実装仕様

### 背景・目的
- 現状のルールベース処理は一言一句が合致しないと認識できず、子供が使えない
- 子供の自然な言葉遣い（ひらがな・崩した表現・文脈依存）に対応するため
- AIで正規化・分類してからディスパッチする設計に全面移行する

### アーキテクチャ原則
- [x] 設計決定: B案（AI正規化JSON → ディスパッチャー → ハンドラー）を採用
- [x] 設計決定: 2回のAI呼び出し構成（①正規化/軽量モデル、②査定/高品質モデル）
- [x] 設計決定: `confidence:low` のとき1回だけ確認メッセージを挟む
- [x] 設計決定: 親専用コマンドはルールベース維持（フォーマットが明確なため）
- [x] 実装: `intent_normalizer.py` 新規作成

### 実運用診断ログ
- [x] `runtime_diagnostics.jsonl` を `log_dir` 配下に追記する
  - ① 対象ユーザー補正、Gemini分類結果、pending処理、査定/雑談のGemini返答抜粋を残す
  - ② 実運用で「なぜその返答になったか」「どこで会話が噛み合わなかったか」を後から分析するため
  - ③ 入力本文と返答抜粋を含むため、公開・外部共有はしない
- [x] 違和感タグを付与する
  - ① `gemini_low_confidence`, `money_related_but_intent_none`, `reply_asks_clarification`,
    `parent_message_used_author_context`, `parent_message_used_child_channel_context`

  - ③ タグは機械判定の目印であり、最終判断はログ本文・返信文・前後文脈で行う

### 運用異常対策（修正提案1〜6）
- [x] 1. 異常残高データを復旧する
  - ① Discord ID級の巨大数値が残高に入った場合は、直前の正常残高へ `data_recovery` として補正する
  - ② 現在残高が壊れると以後の残高確認・査定・目標達成判定が全て不正になるため
  - ③ 既存ログは削除せず、補正ログを台帳に追記して監査可能にする
- [x] 2. 親IDと子供IDの重複による誤ルーティングを防ぐ
  - ① `find_user_by_discord_id` は親→子供の順で解決し、親IDを子供として扱わない
  - ② 親が子供扱いされると、親の自然文が子供向け査定・初期設定・雑談に流れるため
  - ③ 重複IDは `runtime_diagnostics.jsonl` と標準出力に警告を残し、子供操作は代理・チャンネル文脈で行う
- [x] 3. 親を子供フローへ無条件に入れない
  - ① 親の自然文は、代理指定または子供チャンネル文脈で対象子供が1人に特定できる場合のみ子供対象に補正する
  - ② 親は管理者であり、発言者本人の財布管理をする利用者ではないため

  - ① 初期設定、財布チェック、残高報告は `1234円` のように `円` 必須、かつ現実的上限を超えた金額を拒否する
  - ② Discord IDやチャンネルIDのような数字列を所持金として誤確定しないため
  - ③ 貯金目標補完も `円` 必須にして、商品名や型番の数字を金額扱いしない
  - ④ 財布チェックの差分警告は記号を二重表示しないよう整形する

  - ① 例: 「りかのお小遣い金額を変えたい」には `設定変更 りか 固定 300円` 等の形式を案内する
  - ② 金額変更は安全性が重要で、自然文の推測で実行すると誤操作になるため
  - ③ 残高確認のような読み取りは許可し、設定変更・残高調整・支給額変更系だけを誘導対象にする
- [x] 6. 実運用診断ログを追加する
  - ① `runtime_diagnostics.jsonl` に対象解決、Gemini分類、処理ハンドラ、返答抜粋、違和感タグを記録する
  - ② Geminiが悩んだ場面や会話が噛み合っていない場面を後で検出するため
  - ③ 個人情報を含むため外部共有せず、レビュー用の運用ログとして扱う
