作成日: 2026-05-02

## 位置づけ

この文書は、学習シグナル・フォローのヒントが画一的で、買った物や金額への考察が薄いという課題に対する要件再定義である。

重要: ここに記載する「ヒアリングメモ」は、実在の大学・研究機関・教育機関関係者に直接インタビューした結果ではない。サブワーカーによる公開資料レビュー、現行実装レビュー、UX設計仮説を統合したものである。実ユーザーや専門家の発言として扱わない。

## サブワーカー調査メモ

### 公開資料レビュー

- OECD PISA 2022 では、支出判断について親と話す生徒は金融リテラシーが高い傾向が示されている。ただし因果ではなく相関として扱う。
- CFPB は、子どもの金融能力を「実行機能」「金融習慣・規範」「金融知識と意思決定スキル」の3層で捉えている。金額や満足度だけでなく、計画、待つ力、比較、振り返りを扱う必要がある。
- J-FLEC の小学生向け教材は、低学年から高学年へ、使い方・貯め方・お金の流れ・金融トラブル・保険へ段階的に広げている。
- 知るぽるとは、おこづかいを「貯める」「使う」「人のために使う」「増やす」に分ける考え方、月末の振り返り、現金や残高の見える化を重視している。
- Harvard Center on the Developing Child は、計画、目標達成、衝動制御などの中核スキルは実践とフィードバックで育つとしている。
- 親子金融会話の研究では、トップダウンの教示だけでなく、子どもが話し始めた会話や年齢相応の対話、意思決定への参加が重要な要素として示されている。

### 現行実装レビュー

- 現行の `reflection_context` は、過去90日の支出ログ、満足度、理由、財布チェック差分から `dashboard_points` を作る。
- ダッシュボードの `parent_hints` は `reflection_context` から作られておらず、`app/server.py` の簡易分岐で作られている。
- `signals` は `reflection_context` で置き換わるが、`parent_hints` / `child_hints` は簡易集計のままなので、文言が似やすい。
- 金額分析は、合計・金額記録率に偏っている。品目別合計、固定お小遣い比、少額反復、高額低満足、前月比などは見ていない。
- 支出後の理由・満足度補足が別レコードになる場合があり、金額と満足度・理由が同じ購買として結びつかない。
- `ai_follow_policy` は保存と査定プロンプトには使われるが、ダッシュボードのヒント選択にはほぼ使われていない。

## 課題定義

現状の「学習シグナル」「フォローのヒント」は、親にとって次の行動に変換しづらい。

問題は大きく4つある。

1. 文言が画一的
   - 分岐が少なく、同じ家庭・同じ子どもでも似た文が繰り返される。

2. 買った物と金額の意味づけが弱い
   - 「何にいくら使ったか」が、必要性、頻度、本人の満足、目標貯金への影響に接続していない。

3. 親の行動に落ちていない
   - 「理由を書けている」「満足度が低い」だけでは、親が何を聞くべきか判断しづらい。

4. 子どもの成長行動に落ちていない
   - 記録、比較、待つ、予算配分、貯める、振り返る、説明する、再挑戦する、という行動単位に分解されていない。

## 再定義するプロダクト目的

ダッシュボードは「成績表」ではなく、親子会話と次の小さな行動を作るためのコーチング画面にする。

親に出すべきものは、評価ではなく次の4点である。

- 観察: 何が起きているか
- 解釈: どの力を伸ばす機会か
- 問い: 親がどう聞くとよいか
- 次アクション: 子どもが5分以内に何を試せるか

## 新しい分析モデル

### 入力データ

- [x] 支出記録を同一購買単位で識別できる入力に正規化する。
  - ① 新規支出に `entry_id` を付与し、後追い補足を元支出へ紐づける
  - ② 金額、理由、満足度が別レコードに分かれると、買った物と金額の意味づけが浅くなるため
  - ③ 既存ログは読み取り時に暫定マージし、根拠が弱い場合は `source_notes` に残す
  - [x] `entry_id`: 支出記録の一意ID
  - [x] `ts`: 支出日時
  - [x] `item`: 買った物
  - [x] `amount`: 金額
  - [x] `reason`: 買った理由
  - [x] `satisfaction`: 使用後満足度
  - [x] `source`: 通常記録 / 手動支出 / 補足 / 残高調整
- [x] 会話カードの精度を上げるため、支出の意味づけ入力を追加する。
  - ① カテゴリ、必要/欲しい、計画、比較、待機、目標貯金、オンラインリスク、次回行動を保存または推定する
  - ② 金額だけでは、買う前の判断力・比較・待つ力・貯金への影響を評価できないため
  - ③ 最初から全項目を必須にせず、未入力は推定扱いまたは空欄としてカード根拠に明示する
  - [x] `category`: おやつ、文具、本、ゲーム、交通、プレゼント、オンライン等
  - [x] `need_want`: 必要 / 欲しい / 人のため / 学び / 迷い中
  - [x] `planned`: 事前に予定していたか
  - [x] `compared`: 比較したか
  - [x] `waited`: 買う前に待ったか
  - [x] `goal_related`: 目標貯金と関係するか
  - [x] `online_risk`: 課金、広告、個人情報、サブスクなどの注意点
  - [x] `reflection_next`: 次はどうするか

### 分析軸

- [x] 品目・カテゴリの偏りを分析する。
  - ① 同じ品目/カテゴリへの回数、合計金額、固定お小遣い比を見る
  - ② 「よく出る品目」だけでは、金額負担や反復支出の意味が分からないため
  - ③ カテゴリ推定は誤り得るため、詳細表示では推定であることを示す
- [x] 金額と満足度の組み合わせを分析する。
  - ① 高額低満足、高額高満足、少額低満足反復、少額高満足を分ける
  - ② 同じ満足度でも、金額や反復回数によって次に促す行動が変わるため
  - ③ 高額判定は絶対額だけでなく、固定お小遣い比・直近分布・目標貯金への影響を併用する
  - [x] 高額低満足: 次回は買う前に比較・待つ練習へ接続する。
  - [x] 高額高満足: 何がよい判断だったか言語化する。
  - [x] 少額低満足反復: 小さな衝動支出の積み上がりを扱う。
  - [x] 少額高満足: 低コストで満足できる選択を成功例として扱う。
- [x] 計画性を分析する。
  - ① 予定して買ったか、欲しいものリストや貯金目標から買ったかを見る
  - ② お金の使い方を、衝動ではなく目的・計画に接続するため
  - ③ 未入力時は「計画なし」と断定せず、根拠不足として扱う
- [x] 比較行動を分析する。
  - ① 価格、量、品質、代替案を比べた形跡があるかを見る
  - ② 比較できることは、次回の後悔を減らす実践スキルになるため
  - ③ 比較の有無は自由記述からの推定になり得るため、AIの断定表現を避ける
- [x] 予算配分を分析する。
  - ① 使う、貯める、人のため、増やす/学ぶ、の配分を見える化する
  - ② 金額の大小だけでなく、家庭内で話し合える配分にするため
  - ③ 子どもの価値観を否定せず、偏りは次回相談の材料として扱う
- [x] 記録習慣を分析する。
  - ① 記録件数、記録タイミング、理由・満足度・次回行動の充足率を見る
  - ② 記録の質が上がるほど、本人の振り返りと親の伴走に使いやすくなるため
  - ③ 記録漏れは罰ではなく、次の記録方法を決める材料にする
- [x] 親子会話のタイミングを分析する。
  - ① 同じパターンが出た時、目標に影響した時、本人のよい判断が見えた時にカードを出す
  - ② 毎回問いかけると、監視や詰問に見えやすいため
  - ③ 直近の表示履歴を参照し、短期間に同じテーマを繰り返さない

## 新しい出力要件

### 親向け「会話カード」

各カードは次の構造にする。
ここでいう「親の問いかけ」は、実インタビューの質問ではなく、親が子どもへ会話を始めるために画面へ出す1問である。

- [x] 親向け会話カードを、親が次に聞く1問へ変換できる構造にする。
  - ① タイトル、根拠、伸ばしたい力、親の問いかけ、避けたい声かけ、今日の伴走を持たせる
  - ② 親が支出ログを見ても、何を聞けばよいか分からない課題を解くため
  - ③ 親の問いかけは1問だけにし、叱責、人格評価、兄弟比較、禁止一辺倒の表現を避ける
  - [x] タイトル: 例「少額の反復支出」「高額だけど満足度が高い支出」「目標貯金への影響」
  - [x] 根拠: 品目、金額、回数、満足度、期間
  - [x] 伸ばしたい力: 記録、比較、待つ、計画、貯金、説明、安全確認
  - [x] 親の問いかけ: 1問だけ
  - [x] 避けたい声かけ: 叱責、人格評価、兄弟比較、禁止一辺倒
  - [x] 今日の伴走: 一緒にできる5分以内の行動

例:

- 根拠: 「お菓子」が今月5回、合計820円。満足度は平均4.2。
- 伸ばしたい力: 買う前に待つ、少額反復に気づく。
- 親の問いかけ: 「次に買う前、1回だけ“今日じゃなくてもいい？”って一緒に確認してみる？」
- 今日の伴走: 来週のお菓子予算を本人と100円単位で決める。

### 子ども向け「次の小さなチャレンジ」

子どもには親のメモや分析をそのまま出さない。

- [x] 子ども向けには、1回1つの小さなチャレンジだけを出す。
  - ① 5分以内にできる行動を1つ表示し、結果を残せるようにする
  - ② 分析結果を見せるより、子どもが次に動ける形にするため
  - ③ 失敗しても戻れる表現にし、親メモや内部分析を出さない
  - [x] 1回5分以内
  - [x] 1つだけ
  - [x] 結果を残せる
  - [x] 失敗しても戻れる

例:

- 「次にほしいものが出たら、今買う / 待つ / 別のものにする、から1つ選んでみよう。」
- 「同じものを2つ見つけたら、値段以外の違いを1つ書いてみよう。」
- 「今日買ったものについて、よかった点と次に変えたい点を1つずつ書こう。」

### AIフォロー方針の使い方

`ai_follow_policy` は保存して終わりにしない。会話カード選択に反映する。

- [x] `ai_follow_policy` を会話カード選択に反映する。
  - ① `focus_area` をカード優先順位の重みとして使い、親メモは安全な範囲で補助情報にする
  - ② 親が設定した方針と、画面に出る支援テーマを一致させるため
  - ③ 親メモが叱責・比較・人格評価・罰へ寄る場合は、カード選択にもAI返信にも使わない
  - [x] `focus_area=record_habit`: 記録漏れ、記録再開、理由・満足度補足を優先。
  - [x] `focus_area=planning`: 欲しいものリスト、買う前の待機、予算化を優先。
  - [x] `focus_area=satisfaction_reflection`: 高額低満足、低満足反復、成功例の言語化を優先。
  - [x] `focus_area=saving_goal`: 目標貯金への影響、残り期間、今月の貯金額を優先。
  - [x] `focus_area=impulse_spending`: 少額反復、オンライン課金、待つ練習を優先。
  - [x] `focus_area=income_balance`: 入金、固定お小遣い、支出比率、残す金額を優先。

## 実装要件

### Phase 1: 記録データの正規化

- [x] 支出記録に `entry_id` を追加する。
  - ① 新規の支出記録作成時に一意IDを保存する
  - ② 後追いの理由・満足度・チャレンジ反応を同じ購買へ結びつけるため
  - ③ 既存ログには追記でIDを付けず、読み取り時に仮IDまたは推定IDを使う
- [x] 理由・満足度の後追い補足は、別レコードではなく元支出へ紐づける。
  - ① soft pendingで受けた補足を、可能な限り直前の `entry_id` に更新または関連付ける
  - ② 金額と満足度が分離したままだと、金額分析と満足度分析がずれるため
  - ③ 既存の追記方式との互換性を保ち、移行期間は読み取り時マージを併用する
- [x] 既存ログは読み取り時に同一日・同一品目・近い時刻で暫定マージする。
  - ① 金額あり支出と補足行を、同一購買候補として分析時にまとめる
  - ② 過去データを書き換えずに、現行ログからカード根拠を改善するため
  - ③ 推定マージは誤る可能性があるため、`source_notes` に根拠と不確実性を残す

### Phase 2: 分析エンジン再設計

- [x] `reflection_context` を `learning_insights` に拡張する。
  - ① 既存要約は後方互換で残し、新しいカード生成を `learning_insights` に分離する
  - ② 既存ダッシュボードと査定プロンプトを壊さず段階移行するため
  - ③ 旧出力と新出力の対応をテストし、移行中に空表示にならないようにする
- [x] `insight_cards` を返す。
  - ① カード候補を構造化し、UIとAI連携で同じ根拠を使えるようにする
  - ② 文言生成より先に、何を出すべきかを説明可能に決めるため
  - ③ フィールド不足時はカードを出さないか、根拠不足として優先度を下げる
  - [x] `type`
  - [x] `priority`
  - [x] `evidence`
  - [x] `skill`
  - [x] `parent_question`
  - [x] `child_action`
  - [x] `avoid`
  - [x] `next_observation`
- [x] 金額分析を追加する。
  - ① 品目別合計、カテゴリ別合計、固定お小遣い比、前月比、満足度との掛け合わせを見る
  - ② 「何にいくら使ったか」を、次の判断・親子会話・目標貯金への影響に接続するため
  - ③ 金額だけで評価せず、理由・満足度・頻度・目標への影響を合わせて扱う
  - [x] 品目別合計
  - [x] カテゴリ別合計
  - [x] 固定お小遣い比
  - [x] 前月比
  - [x] 高額低満足
  - [x] 少額反復
  - [x] 目標貯金への影響

### Phase 3: ダッシュボードUI変更

親画面:

- [x] 「学習シグナル」ではなく「今週の会話カード」に変更する。
  - ① 親画面の中心表示を、シグナル一覧から会話カードへ移す
  - ② 親が次に何を聞くかへ直接つなげるため
  - ③ 現行サマリーは移行期間中、補助情報として折りたたみに残す
- [x] 各カードに根拠、問いかけ、今日の伴走を出す。
  - ① 初期表示で根拠、親の1問、5分以内の伴走行動を見せる
  - ② 読むだけで会話を始められる状態にするため
  - ③ 根拠が推定の場合は詳細側に注意書きを出す
- [x] 「なぜこのカードが出たか」を表示する。
  - ① 開閉できる詳細に、品目、金額、回数、満足度、理由、判定条件を出す
  - ② 親がカードの妥当性を判断できるようにするため
  - ③ 子ども画面にはこの詳細分析を出さない
- [x] 子どもごとに `ai_follow_policy` とカードの一致度を表示する。
  - ① 親の重視観点とカード種別の対応を示す
  - ② なぜそのカードが優先されたかを説明するため
  - ③ 親メモ本文は子ども向け表示やAI返信に引用しない

子ども画面:

- [x] 「次にやってみること」を1つだけ出す。
  - ① 子ども画面では `child_action` を1つだけ表示する
  - ② 複数の助言を同時に出すと実行しにくくなるため
  - ③ 数字の一覧より前に出すが、残高・目標確認の基本情報は維持する
- [x] 完了/あとで/ちがう、の反応を残せるようにする。
  - ① 子どもの反応を `learning_support_state` に保存する
  - ② 次回チャレンジの重複回避や調整に使うため
  - ③ 反応は評価点ではなく、次回表示の調整材料として扱う
- [x] 親の内部メモは表示しない。
  - ① 子どもテンプレートには親メモ・内部方針・他児情報を渡さない
  - ② 親が安心して方針を書ける一方、子どもに監視感を与えないため
  - ③ テンプレート分岐だけでなく、サーバー側データ構造でも分離する

### Phase 4: AI生成の使い方

- [x] AIに生ログを丸投げしない。
  - ① ルールベースで `insight_cards` 候補を作り、AIには選ばれたカードだけを渡す
  - ② 生ログ自由作文は、根拠の揺れや親メモ露出につながるため
  - ③ AI出力は安全フィルタを通し、直近表示文との類似度を見て同じ問いを繰り返さない
- [x] AIは、選ばれたカードを年齢・口調・親方針に合わせて言い換える。
  - ① 年齢、bot_personality、nudge_strength を使って短く自然な文へ整える
  - ② 同じカードでも子どもの年齢や家庭方針に合わせる必要があるため
  - ③ 金額確定、カード選択、親メモの直接引用はAIに任せない

### Phase 5: フォロー履歴

- [x] `learning_support_state` を追加する。
  - ① 直近カード、直近フォロー時刻、子どもの反応、親の採用/抑制を保存する
  - ② 同じテーマの繰り返しや家庭に合わないカードの再表示を避けるため
  - ③ 永続的な子ども設定とは分け、復元しやすい状態ファイルとして扱う
  - [x] `last_card_type`
  - [x] `last_nudge_at`

  - [x] `last_parent_question`
  - [x] `child_response`

  - ① `last_card_type` と `last_nudge_at` を参照してカードを抑制する
  - ② 子どもが毎回評価される感覚を避けるため

## 能動的な伴走要件

子どもが話しかけてくるまで待つだけでは、記録習慣が切れた時やチャレンジが止まった時に学習機会が失われる。
ただし、能動メッセージは監視や詰問に見えやすいため、1回1通、1アクション、頻度上限つきで扱う。

- [x] 記録空白を見て、子どもに軽く声をかける。
  - ① `no_record_days` 以上、支出記録がない子どもにだけ送る
  - ② 記録が途切れた時に戻る入口を作るため
  - ③ 「放置」「サボり」「ペナルティ」は使わず、「少し空いてるみたい」「名前だけでも」にする
- [x] 未反応の小さなチャレンジを見て、返答しやすい形で声をかける。
  - ① `last_child_action` と `last_nudge_at` があり、一定日数反応がない場合に `やった` / `あとで` / `ちがう` を促す
  - ② 子どもが完璧な報告を書かなくても、次の調整に使える反応を残すため
  - ③ できた/できないの評価にしない
- [x] 成長行動プランの確認日が近い時に、できたことを1つだけ聞く。
  - ① active な `growth_plan.review_at` が近い時だけ送る
  - ② 増額相談を、行動・振り返り・家庭内合意に戻すため
  - ③ 金額確定や支給は親の明示操作だけで行う
- [x] 同じ子どもへ短期間に連投しない。
  - ① `reminder_state.json` に `proactive_child_nudge_last_sent_by_user` を保存し、`min_days_between_nudges` 未満なら送らない
  - ② 声かけが多すぎると説教や監視に感じやすいため
  - ③ 残高監査など別の必須通知とは分けて管理する
- [x] 送信先は子ども本人のチャンネルへ限定する。
  - ① `allow_channel_ids` のメンバーと `discord_user_id` を突合する
  - ② 親チャンネルや共通チャンネルに個別の学習メッセージを出さないため
  - ③ 送信先が見つからない場合は送らない
